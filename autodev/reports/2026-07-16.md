# Autodev Daily Report — 2026-07-16

## TL;DR
Guarded the extension's shared storage layer (`extension/lib/db.js`) against `chrome.storage` failures, closing the last unhardened browser-API surface in the app — the web app got this treatment back in June/July, the extension hadn't.

## Mode
Stabilization (even calendar day, balanced focus policy).

## What Forge built
- `extension/lib/db.js`: wrapped `getApps()` and `saveApps()` in try/catch.
  - `getApps()` now returns `[]` instead of throwing if `chrome.storage.local.get` rejects (extension context invalidated, storage disabled, etc.).
  - `saveApps()` now fails silently instead of throwing if `chrome.storage.local.set` rejects (quota exceeded, disabled).
  - Mirrors the exact pattern already used in `src/lib/storage.js` (`loadJSON`/`saveJSON`) for the web app.
- Root cause: `popup.js`'s and `dashboard.js`'s top-level `render()` calls have no try/catch, so any storage failure previously would have been an unhandled promise rejection — the same blank-screen failure class documented in `CLAUDE.md`, just never fixed for the extension surface. Fixing it once in the shared `db.js` layer covers every caller (`background.js`, `popup.js`, `dashboard.js`, `options.js` via `settings.js`) rather than patching each site.
- Added 2 regression tests in `extension/lib/db.test.js`:
  - `getApps` resolves to `[]` when `chrome.storage.local.get` rejects.
  - `addApp` still resolves `{added: true}` when `chrome.storage.local.set` rejects.
