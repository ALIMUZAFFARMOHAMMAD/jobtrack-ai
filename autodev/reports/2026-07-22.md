# Autodev report — 2026-07-22

## TL;DR
Guarded `extension/lib/settings.js`'s `chrome.storage` calls against unhandled rejections — closing the same class of bug the 2026-07-16 run fixed in `db.js`, just in the sibling shared-storage module used by the Settings page and the Tailor page.

## Mode
Stabilization (even calendar day; app was already healthy going in, so picked from a browser-API-guard audit rather than an explicit backlog item).

## What Forge built
- `extension/lib/settings.js`: wrapped `getApiBaseUrl`, `setApiBaseUrl`, `getResumeText`, `setResumeText` in try/catch, mirroring `extension/lib/db.js`'s existing guard pattern (getters fall back to `""`, setters fail silently and still return the cleaned value).
- Why: `options.js`'s `load()` and `tailor.js`'s `init()` call these functions at module top level with no try/catch. A rejected `chrome.storage.local.get`/`.set` (extension context invalidated, quota exceeded, storage disabled) would throw unhandled — in `tailor.js` this happens *before* the `$("run").addEventListener(...)` line runs, so the Tailor button would end up permanently inert with no error shown to the user. Same failure mode as the bug fixed in `db.js` on 2026-07-16, just one module over that audit didn't reach.
- Added 2 regression tests in `extension/lib/settings.test.js` under a new "storage failure resilience" block, matching `db.test.js`'s existing style.
